# Orbis Arcana State Ownership Inventory (Current Snapshot)

## Purpose

This is an implementation-facing snapshot of runtime state ownership after the recent refactor pass.
It distinguishes:

- Intentional receiver-owned state (acceptable shell/composition ownership)
- Delegated system-owned state (good)
- Remaining receiver-owned migration targets (future)

Use this as a refactor priority map, not a final spec.

## Date Context

- Generated after:
  - input system / gesture / dynamics extraction
  - resources system extraction
  - spell cast executor + handler registry extraction
  - orb runtime pipeline extraction
  - theme + VFX preset SSOT setup
  - event contracts consolidation
  - receiver bootstrap extraction

--------------------------------------------------
1. STATUS LEGEND
--------------------------------------------------

### Status Meanings

- OK (Shell): reasonable to keep in receiver for now
- OK (System): already lives in the right domain/system
- WATCH: acceptable for now, but likely to need formalization
- MOVE NEXT: high-value migration target
- MOVE LATER: future extraction target, lower urgency

### Authority Meanings

- Authoritative: source of truth for gameplay/runtime behavior
- Derived: computed/render-only state
- Runtime Handle: DOM/RAF/timer/audio handles (presentation/runtime plumbing)

8. FUTURE FEATURE READINESS CHECK (SUPER GRACE EXAMPLE)
--------------------------------------------------

Current architecture can support a cross-slice mechanic like "Super Grace" cleanly.

Why
- Orb buoyancy/grace behavior is already orchestrated separately
- Input gesture/dynamics state is in dedicated systems (resettable)
- Spell action execution is modularized
- Resources/spells/world are not trapped in receiver locals

Likely implementation path later (conceptual)
- Trigger action (spell/item/debug) -> spell/action executor
- Apply orb super-grace status / float grace
- Reset input processing state (gesture + dynamics and maybe input runtime caches)
- Optional VFX/UI feedback

This is a sign the schema refactor is working.

--------------------------------------------------
9. CURRENT REFACTOR POSITION (PRACTICAL ASSESSMENT)
--------------------------------------------------

Receiver burden removed (qualitative)
- Input logic/state: largely delegated
- Spell action semantics/execution: largely delegated
- Resources ownership: delegated
- VFX defaults/theme/presets: delegated
- Bootstrap import/hydration: delegated

Receiver burden remaining (mainly)
- VFX runtime implementations
- Orb runtime state object (`physState`)
- UI shell rendering and flow (mostly acceptable)

Practical assessment
- Receiver is now much closer to its rightful role:
  - shell/composer/transport/UI orchestrator
- Remaining work is more about contract hardening and selective runtime module extraction
  than discovering architecture.
