ORBIS ARCANA ARCHITECTURE SCHEMA (WORKING DRAFT v0.3)

Purpose

This document captures the current architecture schema after the recent refactor wave.
It updates v0.2 by marking what is now implemented, what has been hardened, and what remains.

This is still a working architecture document, not a final implementation spec.

--------------------------------------------------
1. CORE DIRECTION (APPROVED, NOW PARTIALLY REALIZED)
--------------------------------------------------

We moved from a prototype structure where `game-receiver.js` did most jobs into a clearer domain split.

Core direction (still approved):
- Receiver is a composition shell and input adapter.
- Systems own gameplay logic and authoritative state transitions.
- Content files define data (spells, world items, configs, themes).
- Effects (VFX/SFX) should become shared runtime modules (game + labs).
- Labs are preview/dev harnesses, not gameplay authority.

Current reality:
- This is no longer just a plan.
- The receiver has been substantially unburdened and many concerns are now delegated.

--------------------------------------------------
2. WHAT IS NOW IMPLEMENTED (MAJOR PROGRESS SINCE v0.2)
--------------------------------------------------

Content and runtime foundations
- Runtime spell content schema separated into content.
- World item content schema separated into content.
- World system supports multiple globes.
- Receiver consumes spell and world-item content schema.

Spell architecture
- `castActionId` routing is implemented and centralized in practice.
- Shared cast action registry exists.
- Spell cast execution policy is modularized (`spell-cast-executor`).
- Spell action handler table construction is modularized (`spell-action-handlers`).
- Canonical spell action IDs normalized (for example `domus_teleport`, `sanctum_shield`).
- `sanctum` is canonical (`sanctus` alias retained).
- `rahta` alias added to all `rota` spells.

Resources ownership
- `resources-system` now owns:
  - energy bank
  - stored globes
- `spell-dispatch-system` no longer owns globe inventory.
- Receiver no longer owns energy bank state directly.

Input architecture
- `input-system` stores normalized latest frame (SSOT).
- `input-gesture-system` owns shake + flat-spin runtime state.
- `input-dynamics-system` owns stability/variability detector state.
- `input-systems-bundle` composes the input systems.
- `input-frame-pipeline` owns ordered per-frame input orchestration.
- `build-input-hud-view-model` owns HUD formatting transform.

Orb runtime architecture (partial)
- `orb-runtime-pipeline` owns per-frame orb runtime orchestration (logic).
- Orb runtime tuning moved to shared content config (`orb-runtime-config-default`).
- `orb-systems-bundle` composes orb gameplay + orb FX systems.
- `physState` is still receiver-owned (intentional for now).

Theme / visual SSOT
- Shared theme token SSOT implemented (`game-theme-default`).
- Shared CSS var adapter implemented (`apply-theme-css-vars`).
- Receiver applies theme on startup.
- VFX lab consumes the same theme SSOT.

Shared VFX preset SSOT (game + lab)
- Shared presets implemented for:
  - bubble shield
  - shockwave
  - flame aoe
  - electric aoe
- Receiver hydrates VFX defaults from shared presets.
- Lab seeds defaults from the same presets.

Bootstrap / composition
- Receiver bootstrap logic split into `src/runtime/receiver-bootstrap.js`:
  - module loading helper
  - bootstrap hydration helper
- Receiver retains orchestration role and injects shell-specific callbacks/state setters.

Contracts / schema hardening
- Event name SSOT added (`src/contracts/events.js`)
- Core systems now consume shared event constants.
- Receiver shell also consumes shared event constants (via runtime hydration map).
- JSDoc interface contracts added for key modules:
  - input-systems-bundle
  - orb-systems-bundle
  - resources-system
  - spell-cast-executor

Orb shatter correctness (previous v0.2 item, still in place)
- Orb shard palette snapshots live computed orb styles at death.

--------------------------------------------------
3. SYSTEM ROLE SPLIT (APPROVED, IMPLEMENTED IN PART)
--------------------------------------------------

Receiver (`game-receiver.js`)
- Receives/translates transmitter data.
- Composes systems and runtime bundles.
- Wires event flow.
- Renders HUD/UI shell and debug readouts.
- Owns shell lifecycle and transport concerns.
- No longer owns many core gameplay/runtime domains it used to own.

Content (`src/content/*`)
- Defines data/config/tuning.
- No gameplay logic.
- Now includes:
  - spells
  - world-items
  - input configs
  - orb runtime tuning
  - theme tokens

Systems (`src/systems/*`)
- Own gameplay state transitions and runtime logic.
- Consume events/state and emit events.
- Now includes dedicated systems/pipelines for input, resources, spell execution, orb runtime orchestration.

Effects (target direction still valid)
- Presets/defaults are already shared.
- Runtime effect implementations still mostly live in receiver and remain a future extraction target.

Labs
- VFX lab now consumes shared theme + effect presets.
- Still acts as preview/dev harness, not gameplay authority.

5. PHYSICS OWNERSHIP DECISION (STILL VALID)
--------------------------------------------------

Decision remains correct:
- Orb motion/physics state stays conceptually within orb for now.
- Physics is behavior/system logic, not a top-level state slice yet.
- World owns environmental physics context.

What changed since v0.2
- Orb runtime per-frame logic has been extracted into `orb-runtime-pipeline`.
- Orb runtime tuning moved to content SSOT.
- The remaining major piece in receiver is the `physState` object itself and RAF shell.

--------------------------------------------------
23. SUMMARY (UPDATED)
--------------------------------------------------

The architecture has progressed significantly beyond the v0.2 planning state. The receiver is now much closer to a true composition shell: input ownership is split across dedicated systems and pipelines, resources ownership has moved into a resources system, spell execution is data-driven and modularized around castActionId, theme and VFX presets are shared SSOTs across game and lab, orb runtime logic is extracted into a pipeline with shared config, and receiver bootstrap/hydration is modularized. Event names are now centralized via a contracts module and key module interfaces have JSDoc contracts. The main remaining work is contract hardening (especially event payload schemas) and selective extraction of remaining large runtime chunks, especially VFX runtime implementations and eventually orb motion state ownership (`physState`).

End of document.
