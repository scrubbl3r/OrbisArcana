ORBIS ARCANA ARCHITECTURE SCHEMA (WORKING DRAFT v0.4)

Purpose

This document is a checkpoint after a major refactor wave.
It updates v0.3 with what is now fully implemented, what has been hardened, and what remains.

This is a working architecture document, not a final implementation spec.

--------------------------------------------------
1. CORE DIRECTION (APPROVED, NOW SUBSTANTIALLY REALIZED)
--------------------------------------------------

Receiver is moving into its intended role:
- composition shell
- input adapter
- event wiring
- UI shell / debug rendering

Delegated domains now live in systems/modules:
- input normalization + gesture + dynamics
- resources
- spell action execution
- orb runtime frame logic + RAF shell
- VFX runtimes (core orb effects)

--------------------------------------------------
2. MAJOR IMPLEMENTED CHANGES (SINCE v0.3)
--------------------------------------------------

Receiver burden reduction (major)
- Core VFX runtimes extracted from `game-receiver.js`:
  - shockwave
  - bubble shield (generic runtime; sanctum-specific behavior remains shell-owned)
  - orb shatter runtime
  - flame aoe
  - electric aoe
- VFX runtimes are now composed through:
  - `src/vfx/effects/vfx-runtimes-bundle.js`

Orb runtime architecture (major)
- `orb-runtime-state` added and now owns orb runtime motion/status state
- `orb-runtime-pipeline` consumes `orbRuntimeState` API (preferred path)
- `orb-runtime-loop` extracted (RAF + dt shell)
- `game-receiver.js` no longer contains the local `physicsStep()` loop
- `game-receiver.js` no longer references `physState` directly
  - `orbRuntimeState` is now the effective SSOT for orb runtime state in receiver

Schema validation feature implemented
- `Super Grace` mechanic added and proven through architecture
  - grants enhanced float grace
  - resets input processing state (input + gesture + dynamics)
- Debug trigger added:
  - `Shift+N` (nirvana test)
- Then upgraded to schema-native action path:
  - `castActionId: "orb_super_grace"`
  - cast action registry metadata
  - spell action handler mapping

Spell chaining schema (new)
- Runtime spells now support chained follow-up actions:
  - `postCastActions: [{ id, payload }]` (new)
  - `postCastActionIds: []` (compat alias still supported)
- `ignis_rota` now uses:
  - `aoe_flame`
  - then `orb_super_grace` with payload `{ ms: 2500 }`

Orb status timing SSOT (new)
- Added `src/content/orb/orb-status-config-default.js`
- Receiver bootstrap now hydrates:
  - default float grace duration
  - domus grace duration
  - super grace default duration

Event payload contract hardening (new pass)
- Expanded payload typedefs in `src/contracts/events.js` for:
  - `voice.spell_cast`
  - `input.shake_triggered`
  - `pickup.collected`
  - `resources.energy_changed` (compat alias: `energy.bank_changed`)
  - `resources.globes_changed` (compat alias: `energy.globe_inventory_changed`)
- Added typed payload annotations at key emit sites in:
  - `world-system`
  - `input-gesture-system`
  - `resources-system`
  - `spell-dispatch-system`

--------------------------------------------------
3. CURRENT SYSTEM ROLE SPLIT (IMPLEMENTATION REALITY)
--------------------------------------------------

Receiver (`game-receiver.js`)
- Transport/plumbing integration
- Shell lifecycle and UI shell rendering
- Composition of systems/bundles/runtimes
- Event subscriptions and orchestration
- Some remaining shell-owned gameplay-adjacent hooks (sanctum-specific shield behavior, palette capture)

Content (`src/content/*`)
- spells (runtime spell schema, cast actions)
- world-items (pickups)
- input configs (gesture, dynamics)
- orb configs (runtime physics/impact + status timings)
- theme tokens

Systems / Runtime Modules (`src/systems/*`, `src/vfx/effects/*`)
- input systems + bundle + pipeline
- resources system
- spell action handlers + executor
- orb runtime state + pipeline + loop
- orb/world systems bundles
- VFX runtimes bundle + effect runtimes

Labs
- VFX lab consumes shared theme and VFX presets
- still a preview/tuning harness, not gameplay authority

5. WHAT IS NOW "DONE ENOUGH" (FOR THIS REFACTOR WAVE)
--------------------------------------------------

Receiver unburdening (core path)
- input path delegated
- spell execution path delegated
- orb runtime frame logic + loop delegated
- VFX runtime implementations delegated
- bootstrap import/hydration delegated

Schema hardening (first serious pass)
- event name SSOT
- key module JSDoc interfaces
- first payload-shape typedef pass
- state ownership inventory v2 (separate doc)

Cross-slice feature validation
- `Super Grace` proves:
  - content -> action registry -> executor -> handler
  - orb + input systems coordination
  - per-spell post-action chaining

--------------------------------------------------
6. WHAT REMAINS (NEXT PHASE)
--------------------------------------------------

High-value hardening (recommended next)
1. Event payload contracts (second pass)
- `voice.spell_loaded`
- `voice.spell_rejected`
- key `orb.*` lifecycle payloads used by receiver/VFX

2. Receiver bootstrap contract cleanup
- reduce setter/hook sprawl where practical
- document bootstrap context more tightly (partially done via JSDoc)

3. VFX runtime lifecycle bundle improvements (optional)
- formal `start/stop/resetAll` behavior in `vfx-runtimes-bundle`
- standardize runtime `getState()` coverage if useful for debug tooling

Future larger work (not urgent)
4. More receiver UI/render extraction (only if it starts to slow iteration)
5. World-items taxonomy split (only when real examples force it)
6. Enemies/level schema expansion (as prototypes emerge)

--------------------------------------------------
7. RECEIVER STATUS (PRACTICAL ASSESSMENT)
--------------------------------------------------

The receiver is now much closer to its rightful place.

It is no longer the main owner of:
- input logic
- resources
- spell action semantics/execution policy
- orb runtime state ownership
- orb runtime frame loop logic
- core VFX runtime implementations

What it still appropriately owns:
- composition
- shell/UI wiring
- transport plumbing
- top-level orchestration

This is a major milestone.

8. CURRENT FEATURE NOTE: SUPER GRACE
--------------------------------------------------

Current behavior
- `ignis_rota` successful cast triggers:
  - flame aoe
  - then `orb_super_grace`
- `orb_super_grace`:
  - grants enhanced float grace
  - resets input processing state (input + gesture + dynamics)

Debug/testing path
- `Shift+N` triggers `orb_super_grace` through the same action executor path

Current tuning sources
- `orb-status-config-default.js` (global default timings)
- `runtime-spells.js` `postCastActions.payload.ms` (per-spell override)

--------------------------------------------------
9. CHECKPOINT SUMMARY
--------------------------------------------------

This refactor wave successfully moved the project from “prototype with architecture intent” to a genuinely modular runtime with delegated systems, shared content-driven tuning, shared VFX presets, and schema-validated cross-slice mechanics.

The next phase should emphasize contract hardening and targeted feature work rather than broad extraction.
