# Receiver Composition

## Purpose

This is a practical map of what `game-receiver.js` still owns vs what is delegated.
It is meant to help future refactors avoid re-centralizing logic into the shell.

--------------------------------------------------
1. RECEIVER'S RIGHTFUL RESPONSIBILITIES (CURRENT)
--------------------------------------------------

Transport / shell plumbing
- receiver connection flow
- LAN / mobile impulse integration
- idle timeout/session disconnect behavior
- pair/connect UI shell

Composition
- creates and wires systems/bundles/runtimes
- subscribes to event bus
- coordinates startup/bootstrap sequencing

UI shell rendering
- HUD DOM updates and shell UI behavior
- debug readout integration
- shell-level keyboard shortcuts/dev triggers

Top-level orchestration hooks
- spell cast orchestration entrypoint (delegates to executor)
- orb lifecycle side effects (overlays, some resets)
- sanctum-specific shield behavior glue
- orb palette capture at death (used by shatter runtime)

--------------------------------------------------
4. WHAT IS STILL IN RECEIVER (INTENTIONAL / ACCEPTABLE)
--------------------------------------------------

Intentional shell ownership
- DOM element references (`els`)
- UI rendering code for shell/HUD
- connection/pairing/status UI
- shell keyboard shortcuts (debug/dev convenience)
- top-level reset/death overlay choreography

Gameplay-adjacent shell glue (acceptable for now)
- sanctum-specific shield color lock + size override behavior
- orb death palette capture (`captureCurrentOrbPalette`)
- shell-level orchestration around orb death/revive resets

--------------------------------------------------
5. WHAT IS STILL A POSSIBLE FUTURE EXTRACTION (NOT URGENT)
--------------------------------------------------

Optional future refactors
- more HUD/UI rendering extraction (only if it slows iteration)
- sanctum-specific shield shell glue into a spell/status presentation module
- orb death/revive shell choreography into a session/lifecycle module
- additional event payload guards/normalizers (beyond typedef docs)

--------------------------------------------------
6. RECEIVER GUARDRails (TO KEEP IT CLEAN)
--------------------------------------------------

When adding a feature, ask:

1. Is this gameplay state?
- put it in a system/state owner, not receiver locals

2. Is this spell behavior routing?
- prefer `castActionId` + action registry + handler factory

3. Is this visual tuning?
- prefer theme tokens / VFX presets SSOT

4. Is this VFX runtime behavior?
- prefer `src/vfx/effects/*` runtime module

5. Is this input detection/gesture state?
- prefer input systems/bundle, not receiver helpers

Receiver should remain:
- shell
- composer
- adapter
- UI layer
